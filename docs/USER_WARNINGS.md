## WARNING 1 — AI Agent Executes Real Commands

The AURA AI agent is not a simulation. When running in operator or system mode it **executes
real shell commands** on your device. Any command it issues can modify, delete, or transmit
your files. Understand what AURA is doing before granting elevated permissions.

**Mitigation:**
- Review `etc/aios.conf` and the AURA secure-run denylist before first boot.
- Run AIOS-Lite in a restricted user account or container when testing unfamiliar
  configurations.
- Always review AURA's audit log at `$OS_ROOT/var/log/aura.log` after automated sessions.

---

## WARNING 2 — Bridge Modules Access Connected Devices

The iOS, Android, and Linux/SSH bridge modules are capable of **reading and writing files on
physically connected or networked devices**. Only connect AIOS-Lite to devices you own or
have explicit permission to access.

**Mitigation:**
- Disconnect bridge modules when not in active use (`os-bridge disconnect`).
- Never enable auto-mount for untrusted devices.
- Be aware that mirror paths (`$OS_ROOT/mirror/`) reflect live data from connected devices.

---

## WARNING 3 — External LLM Backends Transmit Data Off-Device

If you configure an external model backend (e.g., a cloud API), **your prompts and any
context provided to the AI shell will be sent over the network** to that third-party service.
This may include personal information, system details, or file contents if pasted into the
shell.

**Mitigation:**
- Use the local llama.cpp integration to keep all data on-device.
- If an external backend is required, review its privacy policy and ensure it complies with
  applicable data-protection laws in your jurisdiction.
- Never paste passwords, API keys, or sensitive personal data into the AI shell when an
  external backend is active.

---

## WARNING 4 — Root and Elevated Privileges Are Dangerous

Some AIOS-Lite features (disk image building, certain kernel-control operations) may require
root or sudo privileges. Running any software as root carries inherent risk. A misconfigured
AI agent with root access can destroy your system irreversibly.

**Mitigation:**
- Do not run AIOS-Lite as root in a production environment.
- Isolate root-privilege operations inside a virtual machine, container, or dedicated test
  device.
- Audit `bin/aios-sys` and `build/build.sh` before executing with elevated privileges.

---

## WARNING 5 — Data Loss Risk During Mirror Operations

Mirror and sync operations performed by `os-mirror` can overwrite or delete data on target
devices if misconfigured. There is **no undo** for overwrite operations completed by the
bridge modules.

**Mitigation:**
- Back up all important data on every connected device before performing mirror write or sync
  operations.
- Test new mirror configurations on non-production data first.

## WARNING 9 — SSH Keys and Credentials

AIOS-Lite does not manage or encrypt SSH keys or other credentials on your behalf. If your
`$OS_ROOT` directory is accessible by other users or processes, credentials stored or used
within it may be exposed.

**Mitigation:**
- Apply strict filesystem permissions (`chmod 700 $OS_ROOT`).
- Use SSH agent forwarding instead of storing private keys inside the `$OS_ROOT` directory.
- Rotate any credentials exposed by a compromise immediately.
